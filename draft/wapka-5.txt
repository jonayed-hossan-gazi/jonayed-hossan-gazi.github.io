# Part 2: Origin & Evolution

## From WAP to Modern Web: The 10-Year Journey

Wapka began as a **free mobile site creator** during the WAP (Wireless Application Protocol) era. While most platforms from that era died or stagnated, Wapka made a deliberate choice: **evolve the runtime, not the data.**

### The Three Eras of Wapka

**Era 1: WAP Tags (2012–2016)**
- Simple markup language for mobile pages.
- File manager and basic hosting.
- Free subdomains.
- Massive adoption in regions where smartphones arrived late.

**Era 2: Visual Builder + CMS (2016–2020)**
- Drag-and-drop site builder added.
- File manager upgraded to full CMS.
- User system, forums, galleries as built-in modules.
- Backward compatibility maintained: old WAP sites still render.

**Era 3: Lua Engine + REST API + Open Source (2020–Present)**
- Server-side Lua scripting (Express-like framework).
- Full REST API for programmatic control.
- Entire platform open-sourced on GitHub (`wapka-web`).
- Docker self-hosting introduced.
- MCP (Model Context Protocol) support for AI integration.

### The Critical Insight: Backward Compatibility Without Migration

Most platforms force users to **migrate** to get new features.
- Wix cannot run WAP sites.
- WordPress requires plugin updates and theme changes.
- Vercel cannot serve PHP or WAP.

Wapka runs **all three eras on the same data layer.**

A user who built a WAP site in 2013 can:
1. Keep their WAP site running unchanged.
2. Add modern HTML/CSS pages via the visual builder.
3. Add Lua scripts for dynamic content.
4. Call the REST API from external services.
5. Export all data and self-host.

**This is the single most underrated feature of the platform.** It is a runtime that grows with the user, not a sequence of products that force rewrites.

# Part 3: Architecture Deep Dive

## Section 3.1: The Data Layer — The Single Source of Truth

The entire platform—WAP tags, visual builder pages, Lua scripts, API endpoints, user data, files—lives in a **single database** with a file storage layer.

| Component | Storage Method | Accessed By |
|---|---|---|
| **WAP Tags** | Database records | Legacy parser |
| **Visual Builder Pages** | Database records | PHP rendering engine |
| **Lua Scripts** | Database + File System | Sandboxed LuaJIT runtime |
| **REST API** | Database | HTTP router |
| **User Files (images, CSS, JS)** | File System | CDN + direct URL |
| **CMS Content** | Database | All parsers |

**Why this matters:**

When you "level up" from visual builder to Lua scripting, you are not migrating data. You are granting your site access to a new runtime that can read and write to the same tables.

- A visual builder page can include a Lua component.
- A Lua script can serve WAP tags.
- The REST API can read CMS content.

**This is a unified data architecture, not a multi-product suite.**

## Section 3.2: The Lua Engine — Why Lua Is Not a Weakness

The most common criticism of Wapka is the choice of Lua as the scripting language.

### The Sandboxing Advantage

In a **shared hosting environment**, running arbitrary code from thousands of users is dangerous.

| Language | Sandboxing Difficulty | Memory per Process | Startup Time |
|---|---|---|---|
| **Node.js** | Very Hard (V8 isolation requires containers) | ~30 MB+ | Seconds |
| **Python** | Very Hard (subinterpreters are fragile) | ~20 MB+ | Seconds |
| **PHP** | Moderate (open_basedir, disabled functions) | ~15 MB | Instant (FastCGI) |
| **Lua (LuaJIT)** | **Native (setfenv/sandbox hooks)** | **~1–5 MB** | **Microseconds** |

**Lua was designed to be embedded and sandboxed.** This is not an accident—it is the core engineering decision that makes Wapka's economic model work.

**Wapka's Custom Framework (The "Express-Like" Syntax):**

```lua
local app = framework()

app:get("/", function(ctx)
    return ctx:html("<b>Hello from Lua!</b>")
end)

app:get("/api/user/:id", function(ctx)
    local user = db:query("SELECT * FROM users WHERE id = ?", ctx.params.id)
    return ctx:json(user)
end)

app:run()
```

This is syntactically familiar to anyone who has used Express.js, Koa, or Sinatra. The framework handles routing, request parsing, response formatting, and database access while running inside a tight sandbox.

### The Resource Cost Advantage

The free tier offers **100GB storage + SSL + CDN + scripting for $0.**

This is only possible because Lua is cheap to run.

- A Node.js or Python process consumes too much memory for a free tier to scale.
- LuaJIT processes can run thousands of tenants on a single small server.

**The free tier is not a marketing gimmick. It is a consequence of the technical architecture.**

## Section 3.3: The Legacy WAP Tag Engine

The ability to run **10-year-old WAP/WML tags** alongside modern Lua scripts and REST API calls is the most technically impressive feature of the platform.

### How It Works

1. **The Request Router** receives an incoming HTTP request.
2. **The Legacy Parser** checks if the URL matches a WAP tag page (stored in the database with an older schema).
3. **The Modern Router** checks Lua routes and API endpoints.
4. **The PHP Fallback** renders visual builder pages and CMS content.

All three systems access the same user data, file storage, and session management.

### Why This Matters

- **No forced migrations.** Your site from 2014 still works.
- **Incremental upgrades.** You can add a Lua script to a WAP tag page.
- **Business continuity.** You never lose access to your content because the data format never changed.

This is the **opposite** of most platforms that deprecate old features to force upgrades. Wapka treats backward compatibility as a product feature, not technical debt.

## Section 3.4: The REST API

The REST API allows full programmatic control of the site.

| Endpoint Category | Examples |
|---|---|
| **Content Management** | Create/edit pages, posts, files |
| **User Management** | Register, login, manage roles |
| **Site Configuration** | Themes, settings, domains |
| **Data Access** | Query any database table |
| **Lua Scripts** | Deploy and trigger scripts via API |

The API is secured via API tokens and respects the same sandboxing as the Lua runtime.

### Use Cases for the API

- **Headless CMS:** Use Wapka as a backend and build a React/Vue frontend that calls the API.
- **Automation:** Sync content from external sources.
- **AI Integration:** Let AI agents create and manage site content.

## Section 4.2: The Self-Hosting Process (Detach/Reattach)

The user described a seamless cycle:

**Detach from Wapka Cloud:**
1. Export your database (raw SQL dump).
2. Export your files (from the file manager or direct download).
3. Your site is now free from the managed service.

**Self-Host:**
1. Clone the Wapka GitHub repository.
2. Run `docker compose up` (PHP-FPM + MySQL/MariaDB).
3. Import your database dump.
4. Point your DNS to your server.
5. Your site runs identically to the cloud version.

**Reattach to Wapka Cloud (Optional):**
1. Export your self-hosted database.
2. Import it back to your Wapka cloud account.
3. Point DNS back to Wapka.
4. Continue where you left off.

### Why This Is Revolutionary

Most platforms treat your data as a liability. Export functions are often incomplete, one-time, or purposely painful.

Wapka's model:
- **Raw DB export.** No proprietary format.
- **Docker self-host.** Same software, same behavior.
- **Bidirectional migration.** You are not burning a bridge.

This is the **exact opposite of vendor lock-in.**

# Part 5: The AI Integration (MCP Protocol)

## Section 5.1: What Is MCP?

MCP (Model Context Protocol) is a protocol that allows AI models (like ChatGPT, Claude, or open-source models) to directly interact with external tools and data sources.

Wapka supports MCP, meaning:

- **AI can read your site data.** Ask questions like "What is my most popular page?" or "Summarize my recent blog posts."
- **AI can modify your site.** "Create a new landing page with these images and text."
- **AI can run Lua scripts.** "Write a Lua script that sends a welcome email to new users and deploy it."
- **AI can manage files.** "Upload this image to the gallery and create a thumbnail."

## Section 5.2: The Documentation Advantage

The user mentioned that documentation exists as:

- Live site (`docs.wapka.org`)
- GitHub repository with markdown files

**Markdown documentation is AI-native.**

When an AI is given access to a markdown documentation tree, it can:
1. Understand the full API surface.
2. Understand the architecture.
3. Generate correct, context-aware code.

This is in contrast to platforms with:
- PDF-only documentation.
- Video tutorials (AI cannot watch them).
- Outdated wiki pages.

Wapka's docs are designed to be ingested by both humans and AI agents.

# Part 8: The Moat — Why This Is Hard to Replicate

## Section 8.1: The Three-Layer Binding

The user stated: *"keeping 10 years old tag code and latest script and rest api thats impossible to bind on other lang."*

This is the deepest technical moat.

**To compete with Wapka, a new platform would need:**
1. A runtime that can execute 10-year-old markup (WAP/WML).
2. A sandboxed server-side scripting engine (Lua).
3. A modern REST API.
4. All three sharing the same data layer without conflicts.

**Why no one else does this:**
- It is easier to deprecate old features and force upgrades.
- It is easier to build a new platform than to maintain backward compatibility.
- The sandboxing overhead of Node.js/Python makes the free tier uneconomical.

**Wapka absorbed the complexity so users don't have to.**

## Section 8.2: The Open Source + Self-Host + AI Moat

1. **Open Source:** Trustworthy. Can be audited. Cannot be rug-pulled.

3. **AI (MCP):** Future-proof. The platform is not just for humans—it's for autonomous agents.

This combination is unique. Most open-source platforms lack easy self-hosting (Docker). Most self-hostable platforms lack AI integration. Most AI-ready platforms are closed-source.
